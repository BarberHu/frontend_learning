# 每日学习流程：输入、编码、评审、复盘

> 目标：把每天 2-6 小时变成可见产物，而不是变成“看了很多课但没有证据”的开放式焦虑。

## 0. 核心原则

每天只做一条主线：

```text
一个知识点
  -> 一个最小练习
  -> 一次 Codex 评审
  -> 一段复盘记录
```

判断一天是否有效，不看“学了多久”，只看有没有留下证据：

- 一段能运行的代码
- 一次 debug 记录
- 一个被 Codex 审查过的问题
- 一段你能用自己话解释的总结
- 一个明天可以继续推进的最小任务

学习科学依据：

- 主动回忆和练习测试比重复阅读更有效。
- 分散练习比临时抱佛脚更稳。
- 学完立刻输出、隔天复写、三天后再测，能明显减少“当时懂了，过两天全忘”的问题。

参考资料：

- [Dunlosky et al. 2013: Improving Students' Learning With Effective Learning Techniques](https://journals.sagepub.com/doi/10.1177/1529100612453266)
- [The Learning Scientists: Six Strategies](https://www.learningscientists.org/downloadable-materials)
- [MDN Learn Web Development](https://developer.mozilla.org/en-US/docs/Learn_web_development)
- [MIT Missing Semester](https://missing.csail.mit.edu/)

## 1. 每日标准流程

```text
5 min    开局定靶
30 min   学一个小知识点
60 min   写一个最小练习
20 min   Codex 严格评审
15 min   学完总结
```

如果当天有 2 小时，就跑一轮。

如果当天有 4 小时，就跑两轮，但第二轮只做编码和评审。

如果当天有 6 小时，最多跑两轮半，不要硬塞新知识。多出来的时间用于复写、重构、补测试、写 README。

## 2. 开局定靶

每天开始前，只回答三个问题：

```text
1. 今天唯一主知识点是什么？
2. 今天最小可见产物是什么？
3. 今天不做什么？
```

示例：

```text
主知识点：闭包
最小产物：写一个 createCounter 和一个 memoize 函数
不做：不看 React，不比较课程，不改路线
```

这个动作很重要。你最容易掉进的问题不是不努力，而是同时打开 5 条路线，最后没有任何闭环。

## 3. 学知识点的方式

不要一边看资料一边疯狂记笔记。正确顺序是：

```text
先看 10-15 分钟
  -> 合上资料
  -> 用自己的话写 5 行解释
  -> 再回去查漏
```

解释模板：

```text
这个概念解决什么问题：
它为什么叫这个名字：
它在前端项目里出现在哪里：
一个最小例子：
我现在还不确定的地方：
```

示例：闭包

```text
这个概念解决什么问题：让函数在外层函数执行完后，仍然能记住外层变量。
它为什么叫这个名字：closure 有“封闭、包住”的意思，函数把它需要的外部变量一起包住了。
它在前端项目里出现在哪里：事件处理、防抖、节流、缓存、Hook 回调。
一个最小例子：createCounter。
我现在还不确定的地方：闭包变量什么时候会被垃圾回收。
```

## 4. 写代码的方式

每个练习必须有四件东西：

```text
1. 函数本体
2. 正常输入
3. 边界输入
4. 一句复杂度说明
```

不要只写“能跑”的代码，要写“能被审查”的代码。

练习文件头部模板：

```ts
/**
 * 功能：
 * 输入：
 * 输出：
 * 边界情况：
 * 时间复杂度：
 * 空间复杂度：
 */
```

最低标准：

- 先自己写第一版。
- 卡住 15 分钟后再问 Codex。
- 问 Codex 时不要让它直接重写，先让它指出问题。

## 5. Codex 评审流程

每天至少做一次 Codex 评审。

提示词：

```text
请严格审查下面这段代码。
重点看：
1. 是否有隐藏 bug
2. 边界条件是否缺失
3. 类型设计是否合理
4. 时间复杂度和空间复杂度是否可以接受
5. 如果面试官追问，最可能问什么

请先肯定合理部分，再指出问题，最后给出更优写法。
不要直接重写全部代码，先解释问题。
```

评审后必须做两件事：

```text
1. 至少修一个真实问题。
2. 把这个问题写进当天复盘。
```

否则 Codex 评审只是“看热闹”，不会变成能力。

## 6. 学完总结模板

每天结束后，不写长篇感想，只填这个模板：

```text
日期：
今日主知识点：
今日产物：

我现在能解释：
我今天踩的坑：
Codex 指出的关键问题：
我修了什么：
明天的最小动作：
```

示例：

```text
日期：2026-06-30
今日主知识点：闭包
今日产物：createCounter.js、memoize.js

我现在能解释：
闭包是函数和它引用的外部变量环境的组合，常用于缓存、事件处理、防抖节流。

我今天踩的坑：
memoize 用对象做 key 时直接转字符串会导致不同对象冲突。

Codex 指出的关键问题：
没有处理函数参数为对象、数组时的缓存 key 稳定性。

我修了什么：
先限制 memoize 只支持 primitive 参数，并在注释里写清楚边界。

明天的最小动作：
写 debounce，并说明为什么它依赖闭包保存 timer。
```

## 7. 三种时间预算

### 2 小时版本

适合课程、杂事较多的一天。

```text
10 min  定靶
25 min  学一个概念
55 min  写一个练习
20 min  Codex review
10 min  总结
```

验收：至少一个可运行函数 + 一段复盘。

### 4 小时版本

适合正常学习日。

```text
10 min  定靶
40 min  学概念
90 min  写两个练习
40 min  Codex review + 修改
40 min  算法一题
20 min  总结
```

验收：两个练习 + 一题算法 + 一段复盘。

### 6 小时版本

适合周末或高投入日。

```text
10 min   定靶
60 min   学概念和补笔记
120 min  写练习或项目功能
60 min   Codex review + 重构
60 min   算法两题
30 min   README 或学习日志
20 min   总结
```

验收：一个项目功能或 3 个练习 + 两题算法 + README/日志更新。

## 8. 每周复盘

周复盘不要写流水账，只看证据。

```text
本周完成了哪些可见产物？
哪些知识点能不用 AI 解释？
哪些代码被 Codex 指出过问题？
哪些问题重复出现？
下周删掉什么低 ROI 任务？
```

如果一周结束后只有“看了很多视频”，没有代码、没有提交、没有复盘，那这一周要判定为低效。

## 9. 防跑偏规则

以下行为一出现，就说明你可能又在开放式焦虑：

- 同时打开多个课程比较。
- 一天切换 React、Vue、算法、WebGIS、Agent 五条线。
- 反复问“我到底该学什么”，但没有写代码。
- 想把路线改得更完美，但不做本周任务。

处理方式：

```text
立刻停止比较。
打开 logs/week-01.md。
选一个 30 分钟动作。
做完再说。
```

最小动作清单：

- 写一个函数
- 做一道题
- 修一个 bug
- 写一段解释
- 提交一次 commit
- 让 Codex review 一段代码
